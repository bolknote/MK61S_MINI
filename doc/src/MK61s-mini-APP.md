# Разработка приложений APP для MK61s

Версия документа: 23.07.2026

Этот документ описывает пользовательские исполняемые файлы `.APP` для
STM32F401CC: исходный код, manifest, стабильный resident API, согласованную
сборку, установку в C5 и запуск.

## Модель APP

`.APP` — единый тип исполняемого контейнера. Поле `kind` внутри заголовка
различает:

- пользовательское приложение `APPLICATION`;
- системные компоненты FOCAL, TinyBASIC и WBMP viewer.

Отдельных расширений `.MOD` или `.MKA` нет. Системные файлы имеют канонические
пути `/System/FOCAL.APP`, `/System/BASIC.APP` и `/System/WBMP.APP`.
Пользовательский APP можно хранить под любым допустимым именем в любом
каталоге C5.

APP не является переносимым ELF. Это образ для общего 20-КиБ SRAM-overlay,
связанный с точным адресом из конкретной resident-прошивки и сжатый ZX0.
64-байтовый заголовок содержит формат, ABI, размеры, точку входа, CRC потока и
образа, а также размер и CRC resident `.bin`.

При установке прошивка потоково проверяет весь контейнер до изменения дерева
C5. При запуске она снова сверяет заголовок и resident, распаковывает образ
непосредственно в overlay, обнуляет BSS и вызывает точку входа. APP из другого
комплекта отвергается; случайный переход по чужим адресам не выполняется.

Из этого следует главное правило: распространяйте и устанавливайте resident
`.bin` и APP только из одного результата сборки. После изменения исходников
прошивки, профиля платы, ключей или toolchain весь комплект нужно пересобрать.

На F401 `MK61_ENABLE_LOADABLE_MODULES=1` включается профилем платы
автоматически. На штатной F411-сборке загрузчик APP выключен, а системные
компоненты остаются во внутренней Flash.

## Каталог приложения и manifest

Исходники APP не нужно помещать в `code/` и не нужно добавлять в
`build_f401_bundle.sh`. Создайте отдельный каталог:

```text
MYAPP/
├── app.mk61
├── main.cpp
├── logic.cpp
└── include/
    └── logic.hpp
```

`app.mk61`:

```text
format 1
name MYAPP
source main.cpp
source logic.cpp
file include/logic.hpp
```

Директивы имеют следующий смысл:

| Директива | Обязательность | Назначение |
| --- | --- | --- |
| `format 1` | одна | Версия синтаксиса manifest. |
| `name NAME` | одна | Basename результата `Apps/NAME.APP`, до 31 ASCII-байта. |
| `source path.cpp` | одна или больше | Единица трансляции, входящая в overlay. |
| `file path` | необязательно | Заголовок или `.inc`, нужный исходникам. |

Пути задаются относительно manifest. Они должны быть ASCII, без пробелов,
обратной косой черты, абсолютного пути, `.` и `..`. Пути сравниваются без
учёта регистра, чтобы один проект одинаково собирался на Linux, macOS и
Windows. Имя APP использует буквы, цифры, `_` и `-`, не начинается с `_` или
`-` и уникально без учёта регистра. Имена `CON`, `PRN`, `AUX`, `NUL`,
`COM1`--`COM9` и `LPT1`--`LPT9` зарезервированы FAT/DOS и отклоняются.
Компилируемый файл нельзя маскировать директивой `file`: все `.cpp` должны
быть явно перечислены как `source`; `.c`, `.cc`, `.cxx`, `.s` и `.ino`
текущий SDK не принимает.

Проверить manifest без Arduino-сборки:

```bash
./tools/build_f401_bundle.sh --check-app-manifests \
  --app-manifest path/to/MYAPP/app.mk61
```

Готовый пример находится в
`examples/loadable-apps/HELLO/`.

## Точка входа и resident API v1

Сборщик автоматически предоставляет приложению только три SDK-заголовка:

- `loadable_app_api.hpp`;
- `loadable_module_abi.hpp`;
- `rust_types.h`.

Внутренние заголовки resident намеренно не копируются. Поддерживаемая граница
между приложением и прошивкой — версионная таблица
`loadable_app::Api`, переданная в `argument0`. Прямой импорт C++-символов
resident не является публичным ABI.

Минимальная точка входа:

```cpp
#include "loadable_app_api.hpp"
#include "loadable_module_abi.hpp"

namespace {

const loadable_app::Api* api;

bool set_api(u32 argument0) {
  const loadable_app::Api* candidate =
      loadable_app::from_argument(argument0);
  if(!loadable_app::compatible(candidate)) return false;
  api = candidate;
  return true;
}

} // namespace

extern "C" __attribute__((used, section(".mk61_module_entry")))
u32 mk61_module_entry(u32 raw_command, u32 argument0,
                      u32, u32, u32) {
  const loadable_module::Command command =
      (loadable_module::Command) raw_command;
  switch(command) {
    case loadable_module::Command::INITIALIZE:
      return set_api(argument0) ? 0 : 1;

    case loadable_module::Command::APPLICATION_RUN: {
      if(!set_api(argument0)) return 1;
      static const char text[] = "HELLO APP";
      api->display_clear();
      api->display_write_utf8(0, 0, text, sizeof(text) - 1);
      return 0;
    }

    default:
      return 0;
  }
}
```

Имя `mk61_module_entry`, C linkage, пять аргументов `u32` и секция
`.mk61_module_entry` обязательны. Указатель API передаётся и при
`INITIALIZE`, и при `APPLICATION_RUN`; приложение должно проверять magic,
версию и размер через `loadable_app::compatible()`.

API v1 содержит:

| Capability | Возможности |
| --- | --- |
| `CAP_TIME` | `millis_ms`, обслуживание фоновых задач, безопасная задержка. |
| `CAP_TEXT_DISPLAY` | Размер экрана, очистка и UTF-8-текст до 63 байт. |
| `CAP_KEYBOARD` | Неблокирующий опрос и ожидание нормализованной клавиши. |
| `CAP_LED` | Состояние и асинхронное мигание штатного LED. |
| `CAP_SOUND` | Проверенный звуковой сигнал и остановка звука. |

Перед вызовом необязательной группы проверяйте соответствующий capability.
Логические значения `loadable_app::Key` не зависят от физической раскладки
mini, Classic или 40th. Неизвестный scan-code сохраняется как
`KEY_RAW_BASE + scan_code`.

API v1 намеренно не содержит записи C5, низкоуровневого SPI, USB, управления
тактированием и прерываниями. Это уменьшает вероятность повредить файловую
систему из обычного приложения.

## Время жизни

После распаковки нового образа resident один раз вызывает `INITIALIZE`. При
каждом открытии того же кэшированного файла вызывается `APPLICATION_RUN`.
Загрузка другого APP перезаписывает overlay; при возврате к первому файлу его
образ загружается и инициализируется заново.

В один момент в SRAM находится один APP. Вызов синхронный и должен вернуться.
Долгий цикл обязан регулярно вызывать `api->service()` либо использовать
`api->delay_ms()`. Нельзя вложенно запускать другой APP.

Статические POD-переменные допустимы. BSS обнуляется при загрузке, но не перед
каждым повторным `APPLICATION_RUN`. Если состояние должно сбрасываться при
каждом запуске, сделайте это в обработчике команды.

## Ограничения и доверие

- `.text`, `.rodata`, инициализированные данные и BSS вместе занимают не более
  20 КиБ;
- контейнер занимает от 64 до 20 544 байт;
- стек общий с resident и не входит в overlay: избегайте больших локальных
  массивов и глубокой рекурсии;
- linker запрещает глобальные конструкторы и деструкторы;
- исключения, RTTI, `malloc` и `new` не входят в поддерживаемый контракт;
- код собирается для Cortex-M4 Thumb, hard-float ABI, с `-Os -flto`;
- оптимальная ZX0-упаковка выполняется на компьютере; распаковщик в resident
  не меняется от уровня сжатия.

APP — нативный код без песочницы и должен считаться доверенным. Минимальный SDK
не даёт ему случайно подключить внутренний C5 API, но намеренный доступ к
регистрам или произвольным адресам нельзя сделать безопасным проверкой
контейнера. Код, который напрямую перенастраивает SPI flash, USB, MPU,
тактирование или прерывания, находится вне поддерживаемой модели и способен
нарушить работу устройства. Для обычных приложений используйте только
`loadable_app::Api`.

## Сборка

Один APP:

```bash
./tools/build_f401_bundle.sh \
  --profile mini-v3-a00 \
  --app-manifest path/to/MYAPP/app.mk61
```

Несколько приложений:

```bash
./tools/build_f401_bundle.sh \
  --profile mini-v3-a00 \
  --app-manifest apps/CLOCK/app.mk61 \
  --app-manifest apps/GAME/app.mk61
```

Число `--app-manifest` не ограничено искусственной константой. Можно также
передать список через `MK61_APP_MANIFESTS`: разделитель `:` на macOS/Linux и
`;` на Windows. Для одного Windows-пути вида `C:\...\app.mk61` разделитель не
нужен.

```bash
MK61_APP_MANIFESTS=apps/CLOCK/app.mk61:apps/GAME/app.mk61 \
  ./tools/mk61-firmware.cmd --mcu f401 --profile mini-v3-a00 --build
```

Переменная наследуется командами `--build` и `--upload`. Host-компилятор C++17
нужен при наличии хотя бы одного System или пользовательского APP. Комплект
только с пользовательскими APP штатно собирается и тогда, когда FOCAL,
TinyBASIC и WBMP viewer выключены.

Сборщик:

1. собирает resident с `-Os`;
2. извлекает реальный адрес overlay из resident ELF;
3. для каждого APP создаёт изолированный Arduino sketch только с SDK и его
   manifest-файлами;
4. компилирует APP с `-Os -flto`;
5. связывает только перечисленные `source`;
6. проверяет секции, точку входа и 20-КиБ лимит;
7. оптимально сжимает образ ZX0 и создаёт привязанный контейнер.

Результат:

```text
binary/mk61s-M-mini-v3-lcd1602-a00-f401/
├── mk61s-M-mini-v3-lcd1602-a00-f401.bin
├── build.flags
├── build.apps
├── System/
│   ├── FOCAL.APP
│   ├── BASIC.APP
│   └── WBMP.APP
└── Apps/
    ├── CLOCK.APP
    └── GAME.APP
```

`System` содержит только включённые compile-time ключами компоненты.
`Apps` является полностью генерируемым каталогом: следующая сборка удаляет
устаревшие пользовательские артефакты, которых больше нет в manifest-списке.
`build.apps` хранит точный список созданных путей.

## Прошивка и установка

Прошить согласованный resident:

```bash
MK61_APP_MANIFESTS=path/to/MYAPP/app.mk61 \
  ./tools/mk61-firmware.cmd \
    --mcu f401 --profile mini-v3-a00 --upload
```

Если включены System APP, после перезапуска откройте на калькуляторе
`Меню → USB-диск` и выполните:

```bash
./tools/mk61-firmware.cmd \
  --mcu f401 --profile mini-v3-a00 --install-apps
```

Этот второй шаг синхронизирует только канонические файлы из `System`: копирует
включённые и удаляет `FOCAL.APP`, `BASIC.APP` или `WBMP.APP`, если
соответствующий compile-time ключ выключен. Остальные файлы C5 он не трогает.
Пользовательские файлы из `Apps` копируются вручную:

1. откройте на MK61s `Меню → USB-диск`;
2. создайте на диске `MK61S C5` каталог `/Apps`, если он нужен;
3. скопируйте APP из того же комплекта;
4. дождитесь завершения копирования и выполните безопасное извлечение диска;
5. только после sync/eject нажмите `ESC` на MK61s.

То же можно сделать через правую панель `tools/mkc.cmd`. APP видны в MKC,
Проводнике и USB-диске как обычные файлы; атрибуты hidden, system и read-only
не устанавливаются. MKC передаёт APP потоково через `fsput` и сверяет размер и
контрольную сумму.

### Что происходит при копировании многих файлов

C5 не резервирует фиксированный массив слотов APP. Количество ограничено общей
ёмкостью flash, node-ID и каталогом; host-тест создаёт 200 APP, а отдельный
VFAT-тест одним sync импортирует 25 APP.

Одна незавершённая USB-транзакция может содержать до 384 уникальных
512-байтных блоков staging, то есть 192 КиБ изменённых данных и метаданных.
Это предел одной порции до `SYNCHRONIZE CACHE`, а не предел числа хранимых APP.
Если хост пытается превысить его без sync, следующий блок отклоняется без
частичной публикации дерева C5; тест заполняет все 384 позиции, проверяет отказ
385-й и восстановление старых файлов после reboot. Большие наборы
максимальных APP копируйте порциями с безопасным извлечением/sync между ними;
MKC и `fsput` фиксируют файлы последовательно.

USB-копирование большого набора не является одной общей транзакцией: при
отключении питания часть уже завершённых файлов может быть новой, а файлы,
до которых операция не дошла, — старыми или отсутствовать. Это нормальная
семантика пакетного копирования и не означает повреждение файловой системы.

Для каждого публикуемого файла и изменения каталога действуют COW/WAL.
Persistent staging переживает reset; после каждого моделируемого места обрыва
повтор sync доводит пакет до согласованного результата. Новый или изменённый
невалидный APP отклоняется на read-only preflight до удаления, перемещения или
замены любого узла C5. Уже лежащий неизменённый APP не блокирует остальные
операции после обновления resident; его совместимость всё равно проверяется
при запуске. Если FAT повторно выделяет новому файлу освободившийся extent
уменьшаемого APP, C5 атомарно отделяет этот extent после preflight; результат
не зависит от порядка записей каталога. После реального обрыва снова
подключите диск и повторите копирование всего набора.

## Запуск

Короткое `OK` на APP в Проводнике загружает и запускает его. Терминальные
команды эквивалентны:

```text
open /Apps/MYAPP.APP
run /Apps/MYAPP.APP
```

Переименование и перемещение не меняют контейнер. Не помещайте пользовательский
APP под каноническим системным именем в `/System`.

## Диагностика

| Ошибка | Причина или действие |
| --- | --- |
| `app/firmware mismatch` / `PUT_FIRMWARE` | Resident и APP взяты из разных комплектов. |
| `corrupt app` / `PUT_APP` | Неверный заголовок, размер, ZX0-поток или CRC; повторите копирование правильного файла. |
| `app not installed` | Путь не ведёт к совместимому APP нужного kind. |
| `another app is active` | Выполняется вложенная попытка загрузить APP. |
| `undefined reference` | APP вызвал функцию вне API либо не перечислен нужный `source`. |
| `MK61 module does not fit the SRAM overlay` | Image + BSS превышают 20 КиБ. |
| `unsupported global constructors/destructors` | Уберите нетривиальные глобальные объекты. |
| Packer сообщает slot limit | Сжатый контейнер превышает 20 544 байта. |

Перед публикацией сохраните вместе исходники, manifest, весь каталог комплекта
и версию STM32 toolchain. Самостоятельный `.APP` без согласованного resident
`.bin` не является воспроизводимым релизом.
