# Монохромные изображения WBMP

Версия документа: 21.07.2026

MK61s открывает стандартные файлы Wireless Bitmap (WBMP) Type 0 с расширением
`.wbmp`. Это однобитный несжатый формат: строки идут сверху вниз, старший бит
байта расположен слева, `1` означает белый пиксель, `0` — чёрный. Реализация
следует разделу 6 и приложению A спецификации WAP WAE от 24.05.1999:

<https://wapforum.org/what/technical/SPEC-WAESpec-19990524.pdf>

Собственного magic-номера и собственного варианта BMP у просмотрщика нет.
Тип файла в C5 называется `IMAGE1`, но байты файла остаются обычным WBMP и
открываются стандартными программами.

## Подготовка изображения

ImageMagick создаёт совместимый WBMP штатной командой:

```bash
magick input.png -colorspace Gray -threshold 50% -monochrome WBMP:output.wbmp
```

Порог можно менять, например `-threshold 42%`. Для схем и пиксельной графики
полезно заранее привести картинку к целевому размеру без сглаживания:

```bash
magick input.png -filter point -resize '192x64>' \
  -colorspace Gray -threshold 50% -monochrome WBMP:output.wbmp
```

Файл копируется на USB-диск MK61s как обычный `output.wbmp`, в том числе в
произвольный каталог. Его можно открыть коротким `OK` в Проводнике либо из
терминала:

```text
open /Pictures/output.wbmp
```

Каноническое видимое расширение — `.wbmp`. В короткой FAT 8.3-записи оно
представлено как `.WBM`; этот псевдоним также принимается при чтении тома без
LFN.

Встроенных демонстрационных изображений в прошивке нет.

## Отключение просмотрщика при сборке

Просмотрщик включён по умолчанию. Чтобы исключить декодер и экранные алгоритмы
из конкретной прошивки, задайте флаг компилятора:

```text
-DMK61_ENABLE_WBMP_VIEWER=0
```

Тот же параметр можно постоянно изменить в `code/config.h`. Допустимы только
значения `0` и `1`; ошибочное значение останавливает сборку. Отключение касается
только просмотра: C5 и USB-диск продолжают хранить и показывать `.wbmp`, поэтому
файлы не пропадают и снова открываются после установки полной прошивки.

## Ограничения и проверка файла

Максимальный размер WBMP-файла — 1600 байт. Этого достаточно для полного кадра
192x64: 1536 байт пикселей и 4--5 байт заголовка. Остальные типы файлов C5
сохраняют квоту 1536 байт. Изображение может быть шире или выше экрана, если
его общий файл укладывается в квоту; тогда доступна прокрутка.

Перед показом проверяются:

- TypeField `0`, отсутствие extension headers и нулевые reserved-биты
  FixedHeaderField (два разрешённых стандартом type-бита не ограничиваются);
- корректная big-endian multi-byte запись ширины и высоты;
- ненулевые размеры и отсутствие арифметического переполнения;
- точное соответствие длины файла вычисленному растру;
- нулевые неиспользованные биты в конце каждой неполной строки;
- CRC записи C5 при чтении из SPI NOR.

Повреждённый или не относящийся к Type 0 файл не выводится частично.

## Управление

`ESC` и `OK` закрывают просмотрщик. Стрелки влево/вправо сдвигают окно на пять
пикселей. Сдвинутые стрелки влево/вправо перемещают его вверх/вниз на 14
пикселей. Координаты всегда ограничиваются границами изображения, поэтому за
последней строкой или колонкой пустая область не прокручивается.

На LCD1602 клавиши `Г` и `Р` соответственно увеличивают и уменьшают частоту
временного мультиплекса. Настройка действует до закрытия текущего изображения.

## LCD1602: окно 80x14

Символьные LCD1602 физически имеют 32 знакоместа по 5x7 видимых точек и только
восемь слотов CGRAM. Просмотрщик заранее строит до четырёх временных кадров для
всего окна 80x14:

1. Совпадающие остаточные узоры выбираются как общие CGRAM-глифы и повторно
   используются во всех совместимых знакоместах.
2. Пустые знакоместа не занимают слот; одинаковые фрагменты также не
   дублируются.
3. Выбранные точки вычитаются, затем следующий кадр уточняет только остаток.
4. Каждый требуемый пиксель зажигается ровно в одной фазе, а ложные пиксели не
   зажигаются ни в одной. Поэтому одинаковые точки не становятся ярче других.

Последовательность полностью вычисляется при открытии и после прокрутки;
горячий цикл только передаёт готовые кадры. Драйвер читает busy flag
HD44780-совместимого контроллера после каждого байта, а при неподключённой линии
RW автоматически возвращается к безопасным задержкам. Между фазами
сравниваются прежние CGRAM и DDRAM: неизменившиеся слоты и знакоместа не
передаются, соседние изменения объединяются в непрерывные пачки. Экран кратко
гасится только на время фактического изменения видимой CGRAM.

Алгоритм использует исключительно CGRAM и поэтому не зависит от набора знаков
CGROM; один путь работает в сборках LCD1602 A00 и A02.

## UC1609: окно 192x64

Для графического экрана растр сначала целиком декодируется в off-screen буфер
192x64 в нативном page-major формате UC1609. После успешной подготовки все
1536 байт передаются одной SPI-транзакцией без предварительной очистки экрана.
При прокрутке старый кадр остаётся видимым до полной готовности нового.

На время просмотра дисплей переходит в modal fullscreen-режим: фоновые
обновления текстовой сетки и курсора продолжают обслуживать остальную систему,
но не могут перерисовать изображение. При выходе режим снимается и обычный
экран восстанавливается одной полной перерисовкой.

## Использование общей RAM

Файл один раз читается из C5 в `shared_scratch`; повторных чтений SPI NOR и
повторной проверки CRC в каждом кадре нет. Декодированный кадр или готовая
LCD1602-последовательность размещаются во взаимоисключающем
`language_workspace`, который уже используется FOCAL, Tiny BASIC и USB-диском.
Просмотрщик не может работать одновременно с этими режимами, поэтому отдельные
крупные глобальные массивы не выделяются.

Единственное увеличение крупного общего буфера — 64 байта: `shared_scratch`
увеличен с 1536 до 1600 байт ради полной картинки 192x64. С учётом небольшого
состояния драйвера суммарный измеренный прирост постоянной RAM относительно
предыдущей прошивки составляет 88 байт для LCD1602 и 64 байта для UC1609.
USB-кеш по-прежнему получает из `shared_scratch` ровно три сектора по 512 байт;
оставшиеся 64 байта не создают хвостов во flash и не меняют геометрию C5.
