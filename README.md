# MK61S_MINI

Развитие проекта эмулятора МК-61s - <a href="https://web.archive.org/web/20231130170055/https://gitlab.com/vitasam/mk61s/">gitlab.com/vitasam/mk61s</a>

Основной задачей было, получение бюджетного варианта, легкого в повторении всеми желающими. В связи с чем, печатная плата была ограничена размером 10х10 сантиметров. В качестве микроконтроллера используется готовая отладочная плата "BlackPill". Для сборки устройства не требуется паяльная станция или специфичное оборудование. Навыков пайки начинающего радиолюбителя и паяльника из поднебесной, будет достаточно.

Ну и дополнительно, хотелось создать аппаратную платформу, пригодную для реализации других эмуляторов программируемых микрокалькуляторов или вычислительных систем.

Обсуждение проекта и просто общение любителей программируемых микрокалькуляторов, происходит в группе в Телеграм - https://t.me/mk61s

Программный код проекта основан на работе Феликса Лазарева (https://github.com/fixelsan/emu145), в связи с чем ни один "ЕГГОГ" не пострадал и участники проекта могут во всей красе насладиться особенностями работы микрокалькуляторов "Электроника МК-61".

Для снижения порога входа в проект, исходный код предоставлен для среды разработки Arduino IDE.
Для предложенного исходного кода сформированы также бинарные файлы, которые можно найти в соответствующей папке проекта.

Разработка ведется творческим коллективом, в составе:
- Digitalinvitro (https://github.com/digitalinvitro) - программный код
- Vitasam - (https://github.com/vitasam) программный код
- UN7FGO (http://gengen.ru/) - схема, печатная плата и документация
- Bolknote (https://github.com/bolknote/) - программный код: хранилище программ и проводник, USB-диск/VFAT, терминал и M61-скрипты, языковые режимы и редакторы, шрифты FMK и режимы графического дисплея

На данный момент, актуальной является 3-я версия печатной платы устройства, гербер-файлы которой размещены в соответствующем разделе. 

Устройство в сборе, имеет габариты 99х99х19 мм, но никто не мешает "доработать его напильником" и сделать значительно тоньше. Все возможности для этого имеются.

## Часы RTC

На платах со свободными выводами OSC32 встроенный RTC STM32 работает от
часового кварца LSE и сохраняет ход часов от батарейки на входе VBAT. На mini
V2 вывод PC15/OSC32_OUT занят линией DB7 дисплея, поэтому прошивка использует
внутренний LSI и явно выключает сохранённый в backup domain LSE, освобождая
PC15 после старой прошивки или обычного reset. На остальных платах запуск LSE
ограничен тайм-аутом с безопасным переходом на LSI. Режим LSI менее точен и не
гарантирует ход часов при питании только от VBAT.

Управление доступно в USB-терминале:

```text
date
date YYYY-MM-DD HH:MM:SS
```

До первой установки времени или после потери резервного питания команда `date`
молча завершается. После установки она выводит дату и время в том же формате;
успешная установка подтверждается строкой `DATE OK`. Часовой пояс прошивка не
пересчитывает: вводится нужное местное время.

То же значение можно задать на самом устройстве через
`Настройки` → `Дата и время`. Цифровые клавиши заменяют выбранную цифру,
стрелки перемещают курсор, `OK` проверяет и сохраняет всё значение, `ESC`
выходит без сохранения. Пока RTC не установлен, начальным значением редактора
служит дата и время сборки прошивки; автоматически в RTC оно не записывается.

## Меню сборки и прошивки

Интерактивный конфигуратор запускается из корня проекта:

```bash
./tools/mk61-firmware.cmd
```

На Windows достаточно запустить командный файл — из `cmd.exe`, PowerShell или
двойным щелчком:

```bat
tools\mk61-firmware.cmd
```

Это единый полиглотный лаунчер: Windows запускает через него PowerShell-порт,
а в macOS/Linux тот же файл передаёт управление shell-версии. Обе реализации
лежат в служебном подкаталоге. На Windows лаунчер запускает PowerShell без
профиля и с разрешением выполнения только для этого процесса; системная
execution policy не меняется.

PowerShell-порт поддерживает те же профили, отдельный выбор платформы и экрана,
ключи компиляции, сборку, определение устройства, журнал, progress bar и
DFU-загрузку. Он использует тот же `.mk61-firmware.conf`, поэтому формат
сохранённых настроек у Bash- и PowerShell-версий одинаковый. На Windows 10
DFU-устройство ищется через системный PnP по VID/PID `0483:df11`, а загрузка
выполняется официальным recipe STM32 Core через `arduino-cli` и
STM32CubeProgrammer. Поэтому отдельный `dfu-util.exe` на Windows не требуется.
Путь к `STM32_Programmer_CLI.exe` ищется в `PATH` и стандартном каталоге ST;
если его там нет, меню попросит указать путь и сохранит выбор в локальной конфигурации.
В старой консоли Windows маркеры используют однозначные `>`, `( )`, `(*)`,
`[ ]` и `[x]`; Windows Terminal может использовать ту же безопасную разметку.

Он позволяет отдельно выбрать платформу и экран. Для mini V2/V3 доступны
LCD1602 A00 и A02, а для Classic V2/V3 и 40th — UC1609; несовместимые пары
помечаются и не выбираются. Меню может
собрать `.bin`, перевести уже работающий MK61s в системный DFU командой
терминала и загрузить прошивку. Поиск USB не тормозит старт: он выполняется
по пункту «Найти устройство» и автоматически перед загрузкой. DFU-загрузчик
определяется по `0483:df11`.

`dialog` и `whiptail` не обязательны. Когда их нет, инструмент использует
собственный полноэкранный Bash-интерфейс в сдержанной палитре Norton Commander:
тёмно-синий фон только внутри окна, голубые Unicode-рамки, белый текст и жёлтые
отметки. Интерфейс использует
радиокнопки `◉/○`, чекбоксы `☑/☐`, управление стрелками и progress
bar. При навигации обновляются только две изменившиеся строки, поэтому экран не
мигает. Ничего для интерфейса автоматически не устанавливается.

В пункте «Ключи компиляции» можно независимо включать FOCAL, TinyBASIC,
WBMP viewer, USB-экран, расширенные настройки шрифтов, быстрый вызов Explorer и
CORE математику вместо `libm`. Выбранные значения явно передаются сборке как
`-D`ключи. USB-экран по умолчанию выключен. При выключенном
`MK61_ENABLE_USB_SCREEN` из прошивки удаляются сам режим, его статические буферы
и пункт `Разработка → USB-экран`. Если USB-экран включён в
LCD1602-сборке, вместе с ним компилируются настройки графического
шрифта; пункт настроек виден только в активном USB-режиме.

Работающая прошивка позволяет автоматически распознать Classic V2, Classic V3
и 40th по существующему ответу `ver`. У mini V2/V3 этот ответ не содержит
ревизию платы и вариант CGROM A00/A02, поэтому при первом запуске их нужно
выбрать вручную. Платформа, экран и ключи запоминаются в корневом
`.mk61-firmware.conf`; этот локальный файл добавлен в `.gitignore`.
На macOS и Linux при старте быстро ищется только исполняемый `dfu-util` — USB в
этот момент не сканируется. Проверяются сохранённый путь, `PATH`, Homebrew/MacPorts,
`~/Library/Arduino*/packages/STMicroelectronics/tools/STM32Tools/*/macosx/dfu-util`
на macOS, а также системные, Linuxbrew и
`~/.arduino*/packages/STMicroelectronics/tools/STM32Tools/*/linux/<arch>/dfu-util`
на Linux. Если ничего не найдено, меню просит полный путь и сохраняет его в том же
конфиге.

Для воспроизводимой сборки нужны `arduino-cli`, STM32 Arduino Core `2.12.0`,
`LiquidCrystal 1.0.7` и `STM32duino RTC 1.9.0`. Их состояние видно в пункте
«Проверить зависимости». Пункт «Установить Arduino-зависимости» устанавливает
только Arduino Core и эти две библиотеки; сам `arduino-cli` и TUI-утилиты он не
ставит.
На Windows дополнительно требуется STM32CubeProgrammer: `arduino-cli upload`
запускает его через штатный recipe STM32 Core. На macOS/Linux загрузка напрямую
выполняется утилитой `dfu-util`, входящей в пакет STM32 Tools.

Есть и неинтерактивный режим:

```bash
./tools/mk61-firmware.cmd --list-profiles
./tools/mk61-firmware.cmd --profile mini-v3-a00 --build
./tools/mk61-firmware.cmd --profile classic-v3 --upload
./tools/mk61-firmware.cmd --detect
./tools/mk61-firmware.cmd --show-config
```

У PowerShell-порта те же параметры, например:

```powershell
.\tools\mk61-firmware.cmd --profile mini-v3-a00 --build
.\tools\mk61-firmware.cmd --detect
.\tools\mk61-firmware.cmd --show-config
```

### Комплект для STM32F401CC

Для F401CC resident-прошивка и модули должны собираться вместе: адреса вызовов
и CRC точной прошивки входят в каждый `.MOD`. Готовый согласованный комплект
создаёт отдельная команда:

```bash
./tools/build_f401_bundle.sh --profile mini-v3-a00
```

Она помещает в `binary/mk61s-M-mini-v3-lcd1602-a00-f401/` прошивку `.bin` и
включённые ключами `FOCAL.MOD`, `BASIC.MOD`, `WBMP.MOD`. Resident собирается с
`-Os`, а каждый модуль отдельно с `-Os -flto` и затем оптимально сжимается ZX0.
Значения `MK61_ENABLE_FOCAL`, `MK61_ENABLE_TINYBASIC` и
`MK61_ENABLE_WBMP_VIEWER` берутся из одноимённых переменных окружения; `0`
убирает и код интерфейса, и соответствующий артефакт.

После прошивки `.bin` файлы `.MOD` копируются в корень C5 через обычный
USB-диск. Это обычные заменяемые файлы без установленных атрибутов: новую
согласованную версию можно скопировать поверх старой, а ненужный модуль
удалить. Перед установкой прошивка проверяет тип,
размер, CRC контейнера и совпадение resident-прошивки; повреждённый или чужой
модуль рабочую копию не заменяет.

На `ARDUINO_BLACKPILL_F401CC` режим автоматически включает
`MK61_ENABLE_LOADABLE_MODULES=1`; явный `-DMK61_ENABLE_LOADABLE_MODULES=0`
оставлен для нестандартной ручной сборки. На F411 поведение и встроенная
компоновка не меняются. Уже отформатированный C5 переформатировать не нужно:
при первом запуске незавершённая старая USB-транзакция сначала штатно
восстанавливается, после чего хвост staging безопасно отдаётся под модули.

## Проверка исходного кода

Host-тесты требуют `clang++` с поддержкой C++17 и запускаются одной командой:

```bash
tests/run_all_tests.sh
```

Для дополнительной проверки границ памяти и неопределённого поведения:

```bash
MK61_TEST_SANITIZERS=1 tests/run_all_tests.sh
```

Тестовый набор проверяет FOCAL, TinyBASIC, математический CORE-бэкенд,
эксклюзивное владение общими runtime/scratch-буферами, декодер WBMP Type 0,
Virtual FAT и реальное
устройство `program_store` через модель SPI flash с инъекцией ошибок записи и
повреждения каталога, определение физической ёмкости с aliasing и ложным
4-байтным режимом, разбор SFDP/4BAIT, а также FMK1, bit-RLE, графическую
текстовую модель и конвертацию FreeType-шрифтов, когда FreeType доступен в
системе.

Референсная сборка прошивки использует STM32 Arduino Core `2.12.0`,
`LiquidCrystal 1.0.7` и `STM32duino RTC 1.9.0`. Драйвер SPI NOR входит в проект
и не зависит от таблицы моделей `SPIMemory`. Точная матрица плат, дисплеев,
распиновки и аппаратных особенностей приведена в
[описании аппаратной платформы](doc/src/MK61s-mini-Hardware.md) и зафиксирована
в [release workflow](.github/workflows/firmware-release.yml).

LCD1602-сборки без флага ревизии предназначены для платы mini V3. Для
`MK61s-mini_v2` обязательно добавляется `-DREVISION_V2`: у V2 отличаются линии
данных LCD и вывод буззера. Прошивка V3 на V2 может
выглядеть как зависшая: экран не инициализируется, а светодиод остаётся включён.
Release workflow отдельно выпускает V2-варианты A00 и A02.

Светодиод PC13 гасится уровнем `LOW` на mini V2/V3 и Classic V3, а уровнем
`HIGH` — на Classic V2 и 40th. У Classic V2 буззер находится на PA0, у
Classic V3 — на PB9. Эти различия задаются профилем платы и проверяются
host-тестами.

Для двух вариантов Classic используются полные профили
`-DMK61_BOARD_CLASSIC_V2` и `-DMK61_BOARD_CLASSIC_V3`; для юбилейной 40TH —
`-DMK61_BOARD_40TH`. Прежняя комбинация
`-DMK61_DISPLAY_UC1609 -DMK61_KEYBOARD_CLASSIC` остаётся алиасом Classic V2,
но для новых сборок лучше указывать профиль платы явно.

Просмотр WBMP Type 0 включён по умолчанию. Его можно исключить из прошивки
флагом `-DMK61_ENABLE_WBMP_VIEWER=0`; хранение `.wbmp` в C5 и USB-диске при
этом остаётся доступным.
